# CCPM — Контекст проекта

## Обзор проекта

**CCPM** (Qwen Code Project Manager) — это навык агента для спецификационно-ориентированного рабочего процесса, который трансформирует управление программными проектами через AI-агентов. Реализует дисциплинированный 5-фазный процесс: **PRD → Epic → GitLab Issues → Parallel Agents → Shipped Code**.

Проект следует открытому стандарту [agentskills.io](https://agentskills.io), что обеспечивает совместимость с множеством AI-харнессов, включая Qwen Code, Claude Code, Codex, OpenCode, Factory, Amp, Cursor и другие.

### Основная философия
- **Каждая строка кода должна ссылаться на спецификацию**
- Контекст хранится в файлах, а не в голове или истории чата
- Параллельное выполнение через интеллектуальную декомпозицию задач
- GitLab Issues служат единственным источником истины для состояния проекта

## Ключевые концепции

### Фазы рабочего процесса

1. **Plan (Планирование)** — Структурированный брейншторминг создаёт PRD в `.qwen/prds/<name>.md`
2. **Structure (Структурирование)** — PRD становится техническим эпиком в `.qwen/epics/<name>/epic.md`, разбивается на файлы задач
3. **Sync (Синхронизация)** — Эпик и задачи становятся GitLab issues с полной трассировкой
4. **Execute (Выполнение)** — Issues анализируются на параллельные потоки работы, агенты запускаются конкурентно
5. **Track (Отслеживание)** — Все операции статуса запускаются через bash-скрипты (мгновенно, без LLM-затрат)

### Соглашения о файлах

Все данные проекта находятся в директории `.qwen/` в **корне проекта пользователя** (не в этом репозитории):

- **PRDs**: `.qwen/prds/<feature-name>.md` — Документация продуктовых требований
- **Epics**: `.qwen/epics/<feature-name>/` — Технические эпики с:
  - `epic.md` — Основной документ эпика с архитектурными решениями
  - `<N>.md` — Файлы задач (нумеруются по GitLab issue после синхронизации)
  - `<N>-analysis.md` — Анализ параллельных потоков работы
  - `updates/` — Отслеживание прогресса агентов

### Соглашения об именовании

- Имена фич: kebab-case, нижний регистр (например, `payment-integration`)
- Файлы задач до синхронизации с GitLab: `001.md`, `002.md`, и т.д.
- Файлы задач после синхронизации с GitLab: переименовываются в номер issue (например, `1234.md`)
- Git-ветки: `epic/<name>`
- Worktrees: `../epic-<name>/` (на уровне корня проекта)

## Использование

CCPM активируется, когда AI-агент обнаруживает намерение, связанное с управлением проектами. Пользователи говорят естественно:

| Что говорит пользователь | Действие |
|--------------------------|----------|
| "Я хочу создать X" | Брейншторминг + создание PRD |
| "распарсь PRD X" | PRD → технический эпик |
| "разбей эпик X" | Декомпозиция эпика на задачи |
| "синхронизируй эпик X с GitLab" | Создаются issues, настраивается worktree |
| "начни работу над issue N" | Запуск параллельных агентов |
| "стендап" / "каков наш статус" | Bash-скрипт запускается мгновенно |

## Предварительные требования

- `git` и `glab` CLI (GitLab CLI, аутентифицирован: `glab auth login`)
- GitLab-репозиторий для проекта пользователя

## Важные правила

### Правило «Сначала скрипты»
Для детерминированных операций (чтение/отчёты о состоянии) всегда запускайте bash-скрипты напрямую:
- Быстрое выполнение
- Стабильный вывод
- Нет затрат токенов LLM

Используйте LLM только для работы, требующей рассуждений: написание PRD, анализ параллелизма, запуск агентов.

### Безопасность репозитория
Перед любыми операциями записи в GitLab скрипты проверяют, что репозиторий — НЕ шаблон CCPM:
```bash
if [[ "$remote_url" == *"automazeio/ccpm"* ]]; then
  echo "❌ Нельзя писать в шаблонный репозиторий CCPM."
  exit 1
fi
```

### Правило даты/времени
Всегда получайте реальную текущую дату, никогда не используйте заглушки:
```bash
date -u +"%Y-%m-%dT%H:%M:%SZ"
```

## Технические детали

### Интеграция с GitLab
- Используется `glab` CLI для работы с GitLab Issues
- В GitLab нет концепта sub-issues (как в GitHub). Вместо этого задачи связываются с эпиком через:
  - Общий лейбл: `epic:<name>`
  - Task list в описании эпик-issues: `- [ ] #<task_N> <title>`
- Аутентификация: `glab auth login`
- Создание issues: `glab issue create --title "..." --description-file ...`
- Лейблы создаются при инициализации: `glab label create "epic"`, `glab label create "task"`

### Параллельное выполнение
- Задачи с `parallel: true` запускаются конкурентно через нескольких агентов
- Каждый агент изолированно работает со своим контекстом
- Агенты координируют через Git-коммиты: `Issue #<N>: <description>`
- Основная беседа выступает как дирижёр, не погружаясь в детали реализации

### Локальный vs удалённый режим
Все операции сначала работают с локальными файлами для скорости. Синхронизация с GitLab — явная и контролируемая. Основной рабочий процесс (создание PRD, парсинг эпика, разбивка задач, выполнение) работает без GitLab в «локальном режиме».
